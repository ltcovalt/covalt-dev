In this post, I'm going to review how to create a client-side utility for
retrieving arbitrary values from the server, without the need to create a
bespoke AJAX script include for each query.

ServiceNow best practice is to perform queries asynchronously and only retrieve
required values. The `GlideRecord - Client` API supports async queries, but it
returns the full record. To follow the ServiceNow recommendation, you would need
to create a client-callable script include to return only the needed data for
each query and use `GlideAjax` on the client side to call the script include.
This adds a substantial amount of boilerplate to the client-side script and
results in many bespoke script include being created, increasing maintenance and
the surface area for bugs. This is where a reusable utility can help.

## Basic Structure

As mentioned previously, to invoke a server-side script from a traditional
client script, we need to use an AJAX script include. The script include is what
will actually query the server, format the response, and return it to the
client. We could stop here, and it would solve one of our main problems -
reducing bespoke script include sprawl - but it would still result in a decent
amount of boiler plate in the client script to setup the `GlideAjax` object,
invoke the query, and process the response.

To handle client-side boiler plate, I like to make use of a UI script. You can
think of UI scripts in much the same way as script includes, except they store
reusable client code. We can create a global (gasp!) UI Script that invokes the
AJAX script include and takes care of basic response processing, error handling,
and parsing, while presenting developers with a more concise and simple to use
API over the standard `GlideAjax` API.

<Alert type="note">
	You may have heard that UI Scripts should not be global. While I agree with
	this as a matter of standard, utility code that will be used across many
	client scripts is the perfect use case for global scripts and is exactly how
	global scripts should be used. Don't let overly-dogmatic statements scare
	you away from the right solution - just be sure to understand when it is the
	right tool to use.
</Alert>

Something I've been doing more recently is only including AJAX specific logic in
the client-callable script include. All backend logic goes into a separate
script that is not client-callable. This allows the backend logic to be reused
on the server as well and the AJAX script to be a simplified wrapper that parses
AJAX params, handles input validation, formats the JSON response, etc. while the
server-side script performs the more complex logic. This means the final
solution will require three files.

- Query (Script Include)
- QueryAjax (AJAX Script Include)
- Query (UI Script)

## Server logic

The first step is to build the server-side logic in the `Query` script include.
For brevity, I cut out comments and optional validation from the examples. Check
the full [source code](/docs/servicenow/server/query/source) if you want the
complete, more robust script to copy/paste to your environment.

### Function setup

I'll initially only be building one method - `Query.records()`. More will come
later, but this single method solves 90% of the client-side data retrieval
boiler plate. We'll scaffold the `Query` namespace and the `records` method.

```js
const Query = {
	records({ table, query, columns, limit, orderBy, orderByDesc }) {
		// todo
	},
};
```

I set up the records method to take a single parameter object and destructure to
named variables. This allows additional parameters to be added to method later,
without changing the function signature and possibly breaking existing calls to
the method. We'll use these parameters to craft the `GlideRecordSecure` query.

```js
const Query = {
	records({ table, query, columns, limit, orderBy, orderByDesc }) {
		const DEFAULT_ROWS = 100;
		const MAX_ROWS = 1000;
		const MAX_COLS = 20;
		const MAX_DOTWALK_DEPTH = 2;
		limit = limit ?? DEFAULT_ROWS;

		let records = []; // will be used to store the query result
		let gr = new GlideRecordSecure(table);
		if (limit) gr.setLimit(limit);
		if (orderBy) gr.orderBy(orderBy);
		if (orderByDesc) gr.orderByDesc(orderByDesc);
		gr.addEncodedQuery(query);
		gr.query();
		while (gr.next()) {
			// todo - process each record
		}
	},
};
```

Pretty straight forward so far - but take note of the use of `GlideRecordSecure`
here - this ensures that ACLs are applied to the query, so users are only able
to retrieve records and columns they would typically have access to.

<Alert type="critical">
	It is critical to use `GlideRecordSecure` when creating a generic,
	client-accessible interface like this. If not, you run the risk of directly
	exposing an interface that could allow clients to query all data from the
	server. Your information security team will not be pleased.
</Alert>

### Iterate each record

With the function and query setup, we can iterate over each record and push a
plain object per record to our records array. The code below goes inside of the
`while` loop that iterates over the `GlideRecordSecure`.

```js
let record = {};

for (let element of columns) {
	let gElement = gr.getElement(element);
	if (!gElement) continue;
	let value = gElement.getValue() ?? null;
	let current = record;

	// split dotwalked elements and add them to the returned object
	let parts = element.split('.');
	for (let i = 0; i < parts.length; i++) {
		let key = parts[i];
		let dotwalkPath = parts.slice(0, i + 1);
		// set value for the final dotwalk segment
		if (i === parts.length - 1) {
			current[key] = value;
		} else {
			// ensure intermediate objects exist for the dotwalk path
			if (!current[key]) current[key] = {};
			current = current[key];
		}
	}
}
records.push(record);
```

This is where things get a bit more interesting.
`GlideRecord.getElement(element)` supports retrieving dotwalked elements, so our
API can easily support deeply nested values like `user.manager.email`. The final
returned record will represent dotwalked values as nested properties. This means
we only want to assign a value on the final segment (`email` in the above
example), but we need to ensure the intermediate object (`manager = {}`) so that
accessing the nested property doesn't throw an exception.

<Alert>
	Some error handling was stripped for brevity, but the full source code
	handles cases when a scalar value would overwrite an intermediate object or
	when an intermediate object would overwrite a previously assigned scalar
	value.
</Alert>

### Input validation

You can handle (or not) input validation however you like, but I am using my own
[Check](/docs/servicenow/server/check) library. This block can be added just
below the `const` and default value declarations.

```js
// prettier-ignore
let validation = 
    Check({table}).is.string().is.validTable()
	.check({query}).is.string()
	.check({columns}).is.array().has.lengthBetween(1, MAX_COLS)
	.check({limit}).is.a.number().between(0, MAX_ROWS)
	.check({orderBy}).is.an.optional().string().is.oneOf(columns)
	.check({orderByDesc}).optional().string().is.oneOf(columns)
	.result();
if (!validation.ok) {
	return {
		status: 'fail',
		data: validation.errors,
	};
}
```

## Response construction

We could simply return our records array and call it a day, if that's what you
prefer. But as the above input validation section likely gave away, I'm loosely
following the JSend specification for response formatting, as the intent is for
this payload to be passed back to the client. Instead of just returning the
data, or null if none exists or an error occurs, the response is wrapped in a
minimal envelope containing a high-level status, message (if one exists), and
data. See the official [JSend](https://github.com/omniti-labs/jsend) spec for
details.

After the `GlideRecordSecure` while loop, we can return our array of records in
the JSend envelope.

```js
return {
	status: 'success',
	data: records.length > 0 ? records : null,
};
```
